# 飞书多维表格插件：业务根因诊断器

> 本文档由桌面版《飞书多维表格插件：业务根因诊断器.docx》整理录入，便于在仓库内回看与迭代对齐。

## 一、项目概述

本插件旨在解决管理者在复杂业务中的「归因难」问题。当某条业务数据（如：任务未达标、订单未履约）出现异常时，插件能够根据多维表中的关联记录字段，自动逆向追踪数据链路，并结合用户设定的目标条件，精准定位链路中导致异常的「卡点/瓶颈」节点。

## 二、核心概念定义

| 概念 | 说明 |
|------|------|
| **起点记录** | 用户当前选中的那行数据（如：任务 A）。 |
| **追踪维度** | 起点记录中的某个【关联记录】或【查找引用】字段（如：前置依赖）。一条链路只追踪一个维度，避免信息过载。 |
| **链路图** | 以起点记录为叶子节点，沿追踪维度向上展开的树状结构。 |
| **目标条件** | 用户定义的「正常状态」规则（如：状态字段 = 已完成）。 |
| **卡点** | 链路图中，从上往下第一个不满足「目标条件」的节点。 |

**实现注意**：概述中的「追踪维度」包含查找引用；示例实现若仅过滤 `LinkedRecord`，需单独评估是否支持查找引用类型字段。

## 三、核心交互流程 (UX Flow)

### Step 1: 选取起点

- 用户在多维表格中点击某一行（或某行的单元格）。
- 插件通过 `bitable.selection.getSelection()` 获取当前记录 ID。
- 插件顶部显示：「当前诊断：[任务 A]」。

### Step 2: 选择追踪维度

- 插件读取当前表的所有字段，过滤出类型为 `LinkedRecord`（关联记录）的字段列表。
- 渲染为下拉单选框：「请选择追踪维度：[前置依赖 / 所属项目 / 客户]」。
- 用户选择后，插件开始构建链路。

### Step 3: 渲染链路图 (React Flow)

- 从起点记录开始，读取其选择字段中关联的父记录，再递归读取父记录的父记录（限制最大深度为 **5 层**，防止死循环）。
- 使用 React Flow 渲染自上而下的树状图。
- 每个节点显示：记录名称 + 关键状态字段值。

### Step 4: 设定目标条件 (诊断规则)

- 提供配置面板：「目标字段：[状态]」+「目标值：[已完成]」。
- 此配置可保存为**视图级缓存**，下次打开无需重配。

### Step 5: 根因高亮与展示

- 插件遍历链路图中的每个节点，检查其「目标字段」的值是否等于「目标值」。
- **合规节点**：绿色边框。
- **卡点节点**：红色边框 + 闪烁效果，并在节点下方标注原因（例：状态为「延期」，未满足「已完成」）。
- 侧边栏输出文字结论：「诊断结果：任务 A 未达标，根因是上游节点[任务 B]状态为延期。」

**规则对齐**：上文「卡点」定义为从上往下第一个不满足目标条件的节点；分阶段说明 Phase 5 若采用「深度最深的 isBlocked 节点」生成结论文案，需与产品统一为一种规则。

## 四、技术架构与规范（供开发参考）

### 1. 技术栈

| 类别 | 选型 |
|------|------|
| 框架 | React 18 + TypeScript |
| 构建工具 | Vite（遵循飞书插件官方脚手架） |
| 多维表 SDK | `@lark-base-open/js-sdk` |
| 图渲染引擎 | `reactflow`（性能好，易定制节点样式） |
| UI 组件库 | `antd`（飞书插件允许使用，开发效率高） |

### 2. 核心数据结构定义

```typescript
// 链路图中的节点数据结构
interface TraceNodeData {
  recordId: string;
  title: string;         // 节点显示名称（如任务名）
  targetFieldValue: any; // 目标字段的当前值
  isBlocked: boolean;    // 是否是卡点
  reason: string;        // 如果是卡点，原因是什么
}

// React Flow 的 Node 类型
type TraceNode = Node<TraceNodeData>;

// React Flow 的 Edge 类型
type TraceEdge = Edge;
```

### 3. 核心 SDK 调用逻辑

- 获取选中记录：`const selection = await bitable.selection.getSelection();`
- 获取表字段列表：`const fields = await table.getFieldList();`
- 过滤关联字段：`fields.filter(f => f.type === FieldType.LinkedRecord)`
- 读取关联数据示例：

```typescript
// 获取当前记录
const record = await table.getRecord(recordId);
// 读取关联字段的值 (返回的是 recordId 数组)
const linkedRecordIds = record.fields[linkFieldId] as string[];
// 递归获取父记录详情...
```

### 4. 递归追踪算法限制

- **最大深度**：限制递归层数最多 **5 层**。
- **防循环引用**：使用 `Set<recordId>` 记录已访问节点，若发现环状依赖则截断并提示。

### 5. 插件边界与限制（重要）

- **自动化/工作流数据**：如果数据是由自动化从其他表推过来的，且没有使用「关联记录」字段，插件无法追踪来源。只能在 UI 上提示：「该数据由自动化创建，需配置来源字段方可追踪」。
- **公式字段**：暂不主动解析公式内部依赖，仅作为展示节点数据使用。

## 五、分步开发指令（Cursor 分阶段）

建议在 Cursor 中分阶段使用以下方向推进项目：

### Phase 1: 初始化与基础 UI

使用 Vite 初始化一个飞书多维表格插件项目，安装 React、TypeScript、`@lark-base-open/js-sdk`、`antd` 和 `reactflow`。搭建插件的基本布局：左侧为配置面板（选择追踪维度、设定目标条件），右侧为 React Flow 画布。

### Phase 2: 获取上下文与维度选择

编写代码监听多维表的选中事件 `bitable.selection.getSelection`。当用户选中某行时，获取当前表的所有字段，过滤出类型为 `LinkedRecord` 的字段，并在左侧面板生成一个 Antd Select 下拉框供用户选择「追踪维度」。

### Phase 3: 递归构建链路图（核心逻辑）

实现核心递归函数 `buildTraceGraph`。输入当前 `recordId` 和用户选择的 `linkFieldId`，递归读取关联的父记录（限制深度 5 层，防循环引用）。将读取到的数据转换为 React Flow 需要的 Nodes 和 Edges 数组，并在画布上渲染自上而下的树状图。

### Phase 4: 目标诊断与高亮

在左侧面板增加「目标条件」配置：一个字段选择器（仅列出非关联字段）和一个值输入框。用户配置后，遍历 React Flow 的所有 Nodes，检查其 `targetFieldValue` 是否等于目标值。如果不等，将该节点的样式改为红色边框，并更新节点数据中的 `isBlocked=true` 和 `reason` 描述。

### Phase 5: 优化与结论输出

在画布下方或侧边栏底部增加「诊断结论」区域。遍历链路图，找到深度最深的 `isBlocked=true` 的节点（若与「第一个卡点」定义不一致需统一），生成自然语言结论，例如：「诊断结果：当前任务未达标，根因是上游任务 [任务 B] 状态为延期。」

## 六、飞书插件审核与上架规范（代码规范）

> 本节整理自桌面《代码规范.pdf》（插件审核时代码 review 要求：规避数据安全与性能隐患）。搭建工程与提审前请逐条对照。

### 1. 本地打包与产物

- 审核方会**直接静态部署前端产物**（或由你方提供可运行地址），**不做二次构建**；若本地打包失败需先自行解决。
- 在 `package.json` 中设置 **`output`** 为构建产物目录（例如 `"dist"`），部署入口一般为 `dist/index.html`。
- 将 **`.gitignore` 中的 `dist` 条目去掉**，保证仓库内**至少包含**构建产物目录与带 `output` 的 `package.json`；每次代码更新后执行 `npm run build`，再上传**重新构建后的 dist**。

```json
{
  "output": "dist"
}
```

- **静态资源引用必须使用相对路径**，禁止绝对路径。在 `vite.config` 中设置 `base: './'`：

```ts
import { defineConfig } from "vite";
export default defineConfig({
  base: "./",
  // ...
});
```

### 2. 路由

- **禁止使用 history 路由**，请使用 **hash 路由**（避免部署路径下刷新 404 等问题）。

### 3. 上架流程（与审核协作）

1. 本地打包（满足上文产物与路径要求）。
2. 填写共享表单（按飞书开放平台/插件提交流程）。
3. 多维表格审核通过后，会 **Fork 并部署**你的项目。

### 4. 初始化配置（运行时）

插件运行时应能基于当前 Base **遍历数据结构并完成初始化配置**：

- 若**无法得到正确数据结构**，或**必须依赖选中单元格**等前置条件，须在界面给出**明确提示**，引导用户如何操作，避免空白或歧义。

**通用初始化逻辑（推荐）**

- 遍历所有表的字段类型；
- 遍历所有表的记录数；
- 优先选取**符合字段类型要求且记录数最多**的表；
- 若对字段类型无严格限制，则**按记录数**判断；
- 若所有表都没有记录，则**使用第一张表**；
- 按接口返回顺序为插件配置项**匹配合适字段**。

### 5. 监听与响应（数据与选中态）

前端应**实时监听**并在变更时**立即刷新插件状态**，无需用户手动刷新，维度包括：

- `base`、`table`、`view`、`field`、`record`、`cell` 的数据变化；
- **选中状态**变化。

（与本项目「选中行作为起点」强相关，选中切换时必须重算或清空链路，避免展示过期图。）

### 6. 性能

批量读写行记录时，优先使用 **`addRecords`、`setRecords`、`deleteRecords`、`getRecords`** 等**批量接口**，避免在循环中反复调用单次行级接口。

（本项目以只读追踪为主，若后续有批量标注/回写，须遵守此条。）

### 7. 数据安全

除插件功能**必需**的 API 请求外，**禁止将多维表格数据发送到外部**（自有服务器、第三方统计/日志若上传业务数据均不符合要求）。
